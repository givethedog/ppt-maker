## 주요 용어 설명

> 발표에서 자주 등장하는 AI/IT 용어를 먼저 정리합니다.

| 용어 | 영문 | 설명 |
|------|------|------|
| **LLM** | Large Language Model | ChatGPT 같은 대규모 언어 AI 모델. 대량의 텍스트를 학습해 자연어를 이해하고 생성 |
| **에이전트** | Agent | 스스로 판단하고 행동하는 AI 프로그램. 사람의 지시 없이도 목표를 향해 작업 수행 |
| **MCP** | Model Context Protocol | AI가 외부 도구(GitHub, Slack 등)에 접속하는 표준 규격. "AI의 USB-C"라고 비유 |
| **CLI** | Command Line Interface | 터미널/명령줄 환경. 마우스 대신 텍스트 명령어로 컴퓨터를 조작하는 방식 |
| **토큰** | Token | AI가 텍스트를 처리하는 최소 단위. 대략 한글 1글자 또는 영어 4글자 ≈ 1토큰 |
| **RAG** | Retrieval Augmented Generation | AI가 외부 문서를 검색해서 답변에 활용하는 기술. AI의 "참고자료 찾기" |
| **워크플로우** | Workflow | 업무 처리 흐름을 자동화한 것. A→B→C 순서대로 자동 실행되는 파이프라인 |
| **오케스트레이션** | Orchestration | 여러 AI/도구를 지휘·조율하는 것. 오케스트라의 지휘자 역할 |

|------|-----------|-----------|

| **Anthropic** | Claude Sonnet/Opus 4.6 | 코딩 1위, **30h+ 자율 에이전트 작업**, 에이전트 팀 기능 |

### 경쟁의 핵심 축 변화

```
2024: "벤치마크 점수가 높은 모델이 최고"
         ↓

**핵심 트렌드 4가지:**
- 모든 모델이 **추론(thinking) 모드** 탑재 → 기본 기능화
- **에이전트 능력**(장시간 자율 작업)이 새로운 경쟁 지표
- DeepSeek R1 이후 전반적 **가격 하락** → 비용 장벽 소멸

## 3. MCP의 등장과 한계, 그리고 네이티브 AI 시대

### 3-1. MCP란? (빠른 요약)

> 💡 **프로토콜(Protocol)** = 서로 다른 시스템이 소통하기 위한 약속/규격. USB-C가 충전기 규격이듯, MCP는 AI↔도구 간 연결 규격입니다.

**Model Context Protocol** — AI가 외부 도구에 접속하는 범용 표준

```
Before: AI 모델 A × GitHub + Slack + DB = 각각 커스텀 코드 (M×N)
After:  AI 모델 A ─┐              ┌── GitHub MCP Server
        AI 모델 B ─┼── MCP 표준 ──┼── Slack MCP Server
        AI 모델 C ─┘              └── DB MCP Server  (M+N)
```

- 2024.11 Anthropic 발표 → 2025년 OpenAI·Google·MS 모두 채택
- 5,800+ 서버, 300+ 클라이언트, SDK 월 9,700만 다운로드
- 2025.12 Linux Foundation에 기증 → **업계 표준 확정**

### 3-2. 그런데... MCP는 이미 사양의 기미?

MCP가 표준으로 자리잡는 동시에, **주요 서비스들이 자체 AI 에이전트를 내장**하기 시작:

| 서비스 | 자체 AI 통합 | MCP 의존도 |
|--------|-------------|-----------|
| **Notion** | Notion 3.0 AI Agents (2025.09) → 3.3 Custom Agents (2026.02) | MCP도 제공하지만, 네이티브가 주력 |
| **Linear** | GitHub Copilot Coding Agent 네이티브 통합 (2025.10) | MCP 없이 직접 연동 |
| **GitHub** | Copilot CLI + Agent Mode | 자체 CLI가 MCP보다 효율적 |
| **Slack** | AI 워크플로우 빌더 내장 | 자체 자동화 강화 |

### 3-3. CLI vs MCP: 효율성 논쟁

> 💡 **CLI(명령줄 인터페이스)** = 터미널에서 텍스트 명령어로 작업하는 방식. GUI(그래픽 화면)의 반대 개념. 개발자에게는 더 빠르고 자동화에 유리합니다.

> "CLI 도구가 MCP보다 AI 에이전트에게 더 효율적이다" — 2026년 초 개발자 커뮤니티

| 항목 | CLI 방식 (`gh`, `kubectl` 등) | MCP 방식 |
|------|------|------|
| 토큰 비용 | ~200 토큰 | ~55,000 토큰 (GitHub MCP 93개 도구) |
| LLM 친숙도 | 매우 높음 (학습 데이터에 풍부) | 런타임에 처음 만나는 스키마 |
| 효율성 | **33% 우위** (벤치마크) | 컨텍스트 소모 과다 |

### 3-4. MCP의 현재 위치

```
MCP의 미래:
├── 사라지지는 않는다 (이미 표준)
├── 하지만 "만능 열쇠"에서 "여러 방법 중 하나"로 포지션 변화
├── 서비스들은 네이티브 AI 통합 + MCP 서버 병행 ("이중 전략")
└── 진짜 승자: 이 모든 것을 조합하는 "오케스트레이션 레이어" ← ⭐
```

## 4. CLI로의 회귀와 바이브 코딩

### 4-1. AI 코딩 도구의 진화

> 인포그래픽 변환 예정

```
2023  자동완성     → GitHub Copilot, Tabnine    "다음 줄 예측"
2024  대화형 IDE   → Cursor, Windsurf          "채팅으로 코드 수정"
2025  CLI 에이전트  → Claude Code, Aider        "터미널에서 프로젝트 전체 관리"
2026  팀 에이전트   → Claude Code Teams, n8n    "여러 에이전트가 협업"
```

- Claude Code: 2025.05 출시 → **8개월 만에 선호도 1위** (46%)
- 개발자 95%가 주 1회 이상 AI 도구 사용, 75%가 코딩의 절반 이상을 AI와 수행

### 4-2. 바이브 코딩 → 에이전틱 엔지니어링

> 💡 **바이브 코딩(Vibe Coding)** = AI에게 원하는 것만 말하고 코드는 전혀 보지 않는 방식. "분위기(vibe)로 코딩한다"는 뜻에서 유래.

> "바이브에 몸을 맡기고, 코드가 존재한다는 사실조차 잊어라" — Karpathy, 2025.02

**오픈클로(OpenClaw)** — 바이브 코딩의 상징이자 논란
- GitHub 스타 **25만+** (역대 최고속), 오픈소스 AI 비서
- 창시자: "코드를 읽지 않고 배포" → 보안 취약점 2.74배, 설정 오류 75% ↑
- 2026.02 창시자 OpenAI 합류 → 프로젝트 재단 이관

> "바이브 코딩은 이제 지나갔다. 이제는 **에이전틱 엔지니어링**이다." — Karpathy, 2026

**결론:** AI가 코드를 쓰되, 사람이 감독·조율하는 방식이 표준으로 정착 중

## 5. 프롬프트 → 컨텍스트 엔지니어링

> "AI에서 새로운 핵심 역량은 프롬프팅이 아니라 **컨텍스트 엔지니어링**이다"
> — Tobi Lutke (Shopify CEO)

> 💡 **컨텍스트(Context)** = AI에게 제공되는 모든 배경 정보. 프롬프트뿐 아니라 대화 이력, 참고 문서, 사용 가능한 도구 등을 포함하는 넓은 개념입니다.

### 무엇이 다른가?

| | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|--|---------------------|---------------------|
| **범위** | 하나의 텍스트 | 전체 정보 아키텍처 |
| **시점** | 호출 시 1회 | 지속적, 여러 턴에 걸쳐 |
| **초점** | "무엇을 물어볼까" | "AI가 이미 무엇을 알고 있을까" |
| **결과** | 더 나은 문구 | 더 나은 **시스템 설계** |

### 컨텍스트의 7요소

> 인포그래픽 변환 예정

1. **시스템 프롬프트** — AI의 역할과 규칙
2. **사용자 프롬프트** — 현재 과제
3. **대화 이력** — 지금까지의 맥락
4. **장기 메모리** — 세션을 넘는 기억
5. **검색된 정보 (RAG)** — 필요한 문서·데이터
6. **사용 가능한 도구** — API, MCP, 함수
7. **출력 형식** — 구조화된 응답 정의

**핵심:** 컨텍스트 엔지니어링 = "무엇을, 언제, 어떤 형식으로, 얼마나" 가져올지 설계하는 상위 학문. MCP, RAG, Memory는 그 실행 메커니즘.

## 6. 에이전틱 AI 딥다이브: 그래서 뭔데?

### 6-1. 한 문장 정의

> **생성형 AI는 질문에 "답"한다. 에이전틱 AI는 목표를 "달성"한다.**

### 6-2. 쉬운 비유: 식당

| | 생성형 AI (ChatGPT) | 에이전틱 AI |
|--|---------------------|------------|
| **비유** | 레시피를 알려주는 셰프 | 직접 요리하고 서빙까지 하는 셰프 |
| **입력** | "파스타 레시피 알려줘" | "저녁 파티 준비해줘" |
| **출력** | 텍스트 (레시피) | **행동** (장보기→요리→세팅→서빙) |
| **도구** | 없음 | 냉장고, 가스레인지, 카드결제... |
| **판단** | 사람이 다음 단계 지시 | 스스로 다음 단계 결정 |

### 6-3. 에이전틱 AI의 작동 원리

```
┌─────────────────────────────────────────────────────────┐
│                 에이전틱 AI 루프                           │
│                                                         │
│   ① 목표 수신     "월간 매출 리포트 만들어줘"              │
│        ↓                                                │
│   ② 계획 수립     DB 조회 → 분석 → 시각화 → 문서 작성      │
│        ↓                                                │
│   ③ 도구 실행     SQL 쿼리 실행, 차트 생성, 문서 작성       │
│        ↓                                                │
│   ④ 결과 평가     "차트가 불완전하네? 다시 해볼게"          │
│        ↓                                                │
│   ⑤ 반복/완료     문제 해결까지 ③-④ 반복 → 완료 보고       │
│                                                         │
│   핵심: ②~⑤를 사람 개입 없이 자율적으로 수행               │
└─────────────────────────────────────────────────────────┘
```

### 6-4. 핵심 특성 6가지

| 특성 | 설명 | 예시 |
|------|------|------|
| **자율성** | 단계별 지시 없이 스스로 진행 | "리포트 만들어줘" 한 마디로 끝까지 |
| **다단계 추론** | 복잡한 계획을 세우고 실행 | DB→분석→시각화→문서 파이프라인 |
| **도구 사용** | API, DB, 파일시스템, 웹 접근 | Slack에 메시지 전송, SQL 쿼리 실행 |
| **메모리** | 장기·단기 컨텍스트 유지 | 어제 작업한 내용을 오늘도 기억 |
| **적응력** | 실패 시 계획 수정 | API 오류 → 대안 경로 시도 |
| **목표 지향** | 인식→추론→행동 루프 반복 | 결과가 만족스러울 때까지 반복 |

### 6-5. AI의 진화 단계

> 인포그래픽 변환 예정

```
1세대  규칙 기반 챗봇     "FAQ 매칭" (~2022)
  ↓
2세대  생성형 AI          "질문하면 답변" (2023, ChatGPT)
  ↓
3세대  코파일럿           "옆에서 제안" (2024, Copilot)
  ↓
4세대  단일 에이전트       "혼자서 일 처리" (2025, Claude Code)
  ↓
5세대  멀티 에이전트       "팀으로 협업" (2026, Agent Teams)
```

### 6-6. 싱글 에이전트 vs 멀티 에이전트

```

## 7. n8n: 에이전틱 AI를 실현하는 플랫폼

> 💡 **노코드/로우코드(No-code/Low-code)** = 코드를 거의 또는 전혀 작성하지 않고 드래그 앤 드롭 등 시각적 도구로 소프트웨어를 만드는 방식입니다.

### 7-1. n8n이란?

| 항목 | 내용 |
|------|------|
| **정체** | Fair-code 워크플로우 자동화 플랫폼 (2019~) |
| **철학** | "코드의 유연성 + 노코드의 속도" |
| **통합** | 500+ 앱/서비스 연동 |
| **배포** | Self-hosted (무료) 또는 Cloud (€24/월~) |
| **AI** | LangChain 기반 **70+ AI 전용 노드** 내장 |
| **규모** | $2.5B 밸류에이션, Nvidia 투자, 3,000+ 엔터프라이즈 고객 |

### 7-2. 왜 n8n이 에이전틱 AI와 만나는가?

앞서 살펴본 에이전틱 AI의 핵심 조건을 n8n에 대입해보면:

| 에이전틱 AI 조건 | n8n의 대응 |
|-----------------|-----------|
| **자율성** | 트리거 기반 자동 실행 (Webhook, 스케줄, 이벤트) |
| **다단계 추론** | AI Agent 노드 — Plan & Execute 패턴 내장 |
| **도구 사용** | 500+ 통합 + MCP Client/Server + HTTP 노드 |
| **메모리** | Simple / PostgreSQL / Redis 메모리 노드 |
| **적응력** | 조건 분기, 에러 핸들링, 재시도 로직 |
| **목표 지향** | 워크플로우 = 목표 달성까지의 전체 파이프라인 |

- **Plan and Execute Agent** — 계획 후 실행 (에이전틱 AI의 핵심 패턴)

### 7-4. n8n의 포지션: 왜 독보적인가?

```
┌─────────────────────────────────────────────────┐
│              에이전틱 AI 도구 스펙트럼               │
│                                                 │

│                                                 │

│     🔴        🟡       🟡       🟢⭐     🟢    │
│                                                 │
│  유연성 최고                      접근성 최고      │
│  but 개발자 필수                  but 제한적       │

│            n8n = "코드 유연성 + 노코드 속도"        │
│            = Sweet Spot ⭐                       │

```

| vs | n8n의 강점 | n8n의 약점 |
|----|-----------|-----------|
| **vs LangGraph** | 시각적 빌더, 빠른 구축 | 복잡한 상태 머신 제어 열세 |
| **vs CrewAI** | Self-hosted, 데이터 주권 | 정교한 역할 기반 크루 구조 열세 |
| **vs Zapier** | JS/Python 코드 작성 가능, 비용 효율 | 통합 수 (500 vs 7,000) |
| **vs Make** | AI 네이티브 (70+ 노드), 자체 호스팅 | 비기술 사용자 접근성 |

### 7-5. MCP + n8n = 에이전틱 허브

n8n은 MCP를 **양방향**으로 지원 (2025.04~):

```
[외부 MCP 서버들]              [n8n 워크플로우]              [외부 MCP 클라이언트]

                           n8n 워크플로우를 도구로 제공          Cursor, VS Code
```

**→ n8n이 MCP의 "소비자"이자 "공급자" 역할을 동시에 수행**

```
고객 이메일/챗 수신 (Trigger)

  → 시승: CRM에서 가용 차량 조회 → 자동 예약 + 확인 메일
  → AS: 차량 이력 DB 조회 → 예상 비용 안내 → 서비스센터 예약
  → 견적: 옵션 조합 → AI 맞춤 견적서 생성 → 담당 SC에 알림
```

**예시 2: 수입차 시장 인텔리전스 에이전트**
```
주간 스케줄 트리거
  → AI Agent: 경쟁사 가격/프로모션 수집 (웹 검색 도구)
  → RAG: 기존 시장 데이터 DB와 비교
  → 주간 경쟁 동향 리포트 생성
  → Notion에 저장 + Slack 마케팅 채널 공유
```

**예시 3: 서비스센터 예측 정비 에이전트**
```
차량 센서 데이터 수신 (Webhook / API)
  → AI Agent: 이상 패턴 분석 + 정비 필요성 판단
  → 도구: 차량 이력 DB 조회, 부품 재고 확인
  → 고객에게 사전 알림 (SMS/앱 푸시) + 서비스 예약 제안
  → 심각도 판단 후 즉시 입고 권고 or 다음 정기점검 시 처리
```

## 8. 결론: 우리가 지금 할 수 있는 것

### 전체 흐름 요약

> 인포그래픽 변환 예정

```
모델이 똑똑해졌다 (LLM 경쟁)
  → 도구를 쓸 수 있게 됐다 (MCP, 네이티브 AI)
  → 코드도 직접 쓴다 (CLI, 바이브 코딩)
  → 컨텍스트를 스스로 관리한다 (컨텍스트 엔지니어링)
  → 혼자서 일을 한다 (에이전틱 AI)
  → 팀으로 협업한다 (멀티 에이전트)

  ★ 이 모든 것을 시각적으로 조합하고 실행하는 플랫폼 = n8n
```

### 왜 n8n인가?

1. **진입 장벽:** LangGraph는 Python 필수, n8n은 드래그 앤 드롭으로 시작
2. **데이터 주권:** Self-hosted로 민감 데이터가 외부로 나가지 않음
3. **현실적 통합:** 500+ 앱 연동으로 우리의 레거시 시스템과 바로 연결
4. **양방향 MCP:** 외부 도구를 소비하면서 동시에 우리 워크플로우를 도구로 노출
5. **점진적 도입:** 단순 자동화부터 시작 → AI 에이전트로 점진적 확장

### 제안: 다음 스텝

| 단계 | 액션 | 목표 |
|------|------|------|
| **1주차** | n8n Self-hosted 설치, 기본 워크플로우 체험 | 도구 익히기 |
| **2주차** | AI Agent 노드로 간단한 자동화 구축 | 가능성 검증 |
| **3주차** | 실제 업무 하나를 에이전틱 워크플로우로 전환 | 실무 적용 |
| **4주차** | 팀 리뷰, 확대 적용 여부 결정 | Go/No-Go |

### 마무리

```
"에이전틱 AI의 성패는 모델이 아니라 오케스트레이션에 달려 있다."

"성공할지 실패할지는 아직 모른다.
 하지만 해보지 않으면 영원히 모른다."
```
